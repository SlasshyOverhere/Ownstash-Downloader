## 2024-05-31 - Cross-Site Request Forgery (CSRF) via Local Extension Server CORS Bypass
**Vulnerability:** The local extension server (`src-tauri/src/extension_server.rs`) serving on port 47152 was configured with `warp::cors().allow_any_origin()`. This allowed any website to send cross-origin requests. Coupled with an insecure `GET` route (`/download?url=xxx`), this permitted a classic CSRF vulnerability where any website could force the application to download malicious files or launch the app to the foreground by simply embedding an `<img>` or `<iframe>` pointing to the localhost endpoint.
**Learning:** Local servers communicating with browser extensions must strictly validate the origin of requests and should not use permissive CORS policies like `allow_any_origin()`. Furthermore, state-changing operations like starting a download must never be exposed via `GET` endpoints, as they are trivially exploitable by malicious web pages without even needing JavaScript.
**Prevention:**
1. **Restrict CORS:** Use restrictive CORS rules or omit CORS generation so that browsers enforce the same-origin policy, thus failing preflight checks for standard web pages attempting cross-origin POST requests with custom headers.
2. **Require Custom Headers:** Add a requirement for a custom header (e.g., `x-extension-request: true`) to sensitive API endpoints. Standard web pages cannot send custom headers cross-origin without a successful CORS preflight, effectively neutralizing CSRF.
3. **No State-Changing GETs:** Ensure all state-changing endpoints only accept `POST` or other appropriate verbs.
