## 2024-03-20 - Adding ARIA Roles and States to Custom UI Components
**Learning:** This application makes heavy use of custom, interactive UI elements constructed from standard `div` and `button` tags. While visually engaging, these elements (like tabs and segmented controls/filters) inherently lack semantic meaning and keyboard focus indicators for assistive technologies. Relying on visual styling alone to convey state (e.g., active tab) results in an inaccessible experience for screen reader users. Furthermore, interactive elements often lack clear focus outlines (`focus-visible`) for keyboard navigators.
**Action:** When working with custom interactive elements (tabs, filters, toggles) built from basic HTML elements in this project, explicitly implement proper ARIA roles (`role="tablist"`, `role="tab"`, `role="radiogroup"`, `role="radio"`, `role="switch"`), dynamic ARIA states (`aria-selected`, `aria-checked`), provide clear labels via `aria-label` for grouping elements, add `focus-visible` utility classes for keyboard accessibility, and mark purely decorative icons with `aria-hidden="true"`.
